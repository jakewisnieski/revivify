# Decision log

The product and architecture decisions behind Revivify, and why. Each was made deliberately; this is the reasoning trail.

### 1. Target user: non-developers using AI coding agents
People who build with Claude Code / Cursor but can't read the code to judge it. This is the user with the sharpest, least-served version of the "is my AI output any good?" problem — and one the author lives personally.

### 2. Beachhead domain: landing / marketing pages
The single most common thing non-developers generate with AI, with ironclad, citable standards (WCAG, Lighthouse, Search Essentials) and visible, demonstrable violations. Go deep on one domain first; generality comes later via pluggable best-practice packs (the architecture implies it without having to build it).

### 3. Proactive steering over after-the-fact scanning
The competitive scan showed every non-dev quality tool scans *after* a site is built. Steering the agent *during* generation is the uncrowded, defensible space — and published evidence that up-front rules beat no-rules by ~7–14 points supports it.

### 4. AXI-designed CLI + Claude Code hook — not an MCP server
[AXI](https://axi.md/) (design principles for agent-friendly CLIs) benchmarks ~2.3× fewer input tokens and lower cost than MCP for the same job, with no server to run or configure. A Claude Code hook makes the check fire deterministically and gate "done." MCP is deferred to a later adapter for non-Claude agents.

### 5. Deterministic validators for citable authority
Checks wrap **axe-core** and **Lighthouse** — real, industry-standard engines — so every verdict maps to a published rule, not an LLM's opinion. That is what lets a non-developer *trust* the result.

### 6. Check UX designed for people who can't read code
Borrowed and re-aimed from the `no-mistakes` gate: capture the user's **intent** first; triage every finding as **"we'll fix it" / "just so you know" / "your call"**; drive an **own-the-fix loop** (the agent applies fixes, Revivify re-checks); speak **plain language to the human and structured data to the agent.**

### 7. Trust score: 1–10, composed from real sub-scores
Headline `X / 10` shown beside a concrete **"N of 15 checks passing,"** rolled up from Lighthouse's category scores and axe results. Every point maps to a real, explainable increment; non-applicable checks drop out so a simple page isn't penalized.

### 8. Rule prioritization: 15 MVP / 5 fast-follow / 11 roadmap
Ranked by (user impact) × (frequency in AI output) × (deterministically checkable) × (a non-dev can grasp it + the agent can fix it). The MVP deliberately includes every catastrophic-when-present check (e.g. accidental `noindex`, which makes a page invisible to Google). Full tiers in [`rule-catalog.md`](rule-catalog.md).

### 9. Ship-ready bar = a perfect 10/10
The gate clears only at **10/10** — every applicable check passing. A non-developer can't judge which remaining gap is the one that breaks their site, so any bar short of perfect leaves them wondering whether the leftover is the thing that matters. A perfect bar is the only threshold that lets them trust *nothing broken was left behind* — which is the whole promise. Two safeguards keep it fair rather than punishing: **non-applicable checks drop from the denominator** (a simple page isn't held to rules it can't violate), and **"your call" judgment items must be explicitly resolved** — fixed or knowingly accepted — so a deliberate brand choice doesn't trap the page below the bar. The threshold is configurable in `.revivify.yaml` but ships at 10/10.

### 10. Rule set reconciled to what the engines can actually check (M1)
Wiring the real engines (M1) refined the aspirational 15-rule MVP: **13 rules map cleanly to reliable Lighthouse/axe audits and are live.** Two do not have a trustworthy automatic check and are **deferred rather than faked** — **non-text/UI contrast (WCAG 1.4.11)** (axe doesn't detect it reliably) and **broken outbound links / 404s** (no solid Lighthouse audit for a page's own links); both need a purpose-built check in a later milestone. The trust score's denominator is therefore the count of *applicable* rules ("N of M passing"), not a hardcoded 15. Honesty about what's actually verified matters more than hitting a round number — a non-developer trusts the gate only if every green check is real.

### 11. Add a visual cockpit UI — revisiting "CLI + hook only" (decision #4)
Decision #4 chose an AXI CLI + Claude Code hook and put a dashboard out of scope. Dogfooding M1 exposed the gap: the audit is **invisible** to the person it's built for. Headless Chrome shows nothing; the CLI prints a *conclusion* (a number) but not the *process*, so a non-developer is asked to take the score on faith — the exact "trust us" posture the product exists to replace. So we added **`revivify ui`**, a local web-app cockpit that reuses the engine unchanged and **streams the audit live** (Chrome launching, category gauges filling, rules ticking off, the trust dial). It is **additive**: the engine, CLI, and (coming) hook remain the machine-facing core; the UI is the human-facing trust-and-visibility layer, and it must stay the *agentic* check→fix cockpit (not a passive scan-after dashboard, which is the Siteimprove space we differentiate from). Brand note: the UI uses **violet** for brand/action elements and reserves **green** strictly for pass/go signals, so it doesn't read as a clone of green-forward incumbents.

### 12. HTTPS isn't verifiable on a local page — report it not-applicable
Revivify serves the page from `http://127.0.0.1` to audit it, and browsers treat localhost as a secure context, so Lighthouse's HTTPS audit *always* passes locally regardless of the real deployment. A green "Served securely over HTTPS" would be false reassurance — the thing we refuse to do. So HTTPS is marked **not-applicable for local checks** (dropped from the denominator, with a plain note that it's verified once deployed). It becomes a real check when Revivify audits a live URL — a natural future enhancement.

### 13. Adopt an SDLC / GitHub workflow — stop committing to `main` in a pile
Through M0–M2 we shipped good work but did it the wrong way: every commit landed **directly on `main`**, with no issues, branches, PRs, tags, or releases. Conventional Commit messages were the one thing we got right. That's "code in a pile," not a software development lifecycle — and for a product whose entire thesis is *a trustworthy gate that refuses to let unverified work ship*, holding our own repo to a lesser bar is incoherent. So from **M3 forward** we run a real loop — **issue → branch → Pull Request → automated + review gate → squash-merge → tag at milestone boundaries** — mapping each SDLC phase to its native GitHub feature. Jake owns **intent and the gates** (writing issues as outcomes, defining "done," approving merges); Claude runs the **mechanics** (branches, commits, PRs) but never touches `main` without explaining first and never merges or releases without a go-ahead. The load-bearing switch is **branch protection on `main`, turned on before the first feature branch** — that single rule is what makes the process genuine rather than cosmetic, and it's easiest to set before there's work-in-progress to route around. We **do not rewrite the existing `main` history**: it's already pushed, rewriting shared history is risky, and — consistent with the honesty ethos of #10 and #12 — an accurate record ("adopted at M3") is more credible for a portfolio than a fabricated-clean one. Full process contract in [`github-workflow.md`](github-workflow.md); the always-loaded short version for the agent is the repo-root [`CLAUDE.md`](../CLAUDE.md).

### 14. The "done" gate is a Claude Code Stop hook, rolled out warn-then-block (M3)
M3 makes "done" mean *verified*. The gate is a **Claude Code Stop hook** — it fires when the coding agent finishes a turn (the natural "I'm done" moment) and runs `revivify gate`. A Stop hook treats **exit code 2** as "not done, keep going," which is exactly a block: when the page is below the bar, the hook feeds the failing checks back to the agent. Enforcement is **phased on purpose**: it ships **`warn`** (a non-blocking nudge that names what's failing) so it doesn't obstruct while a page is mid-build, and flips to **`block`** (hard-stop until the score clears the bar) as we approach the demo — one line in `.revivify.yaml`. The check *mode follows enforcement*: **`warn` runs the instant static pre-check** (no Chrome, cheap to fire every turn) while **`block` runs the full Lighthouse audit** — the real certifying gate — so tightening the screw also deepens the check automatically. The gate is deliberately **safe as a hook**: no page yet (mid-build) reports "nothing to check yet" and exits 0 rather than crashing the session. This is decision #4's "hook that gates 'done'" made real, and #6's "gate before ship" made automatic.

### 15. `revivify init` is non-destructive by contract — never clobber the user's files
`init` runs inside *someone else's* project, so it treats their files as sacred. Revivify-owned files (`.revivify.yaml`, the `.revivify/` pack) are skipped if present and only regenerated with `--force`; but the project's **`CLAUDE.md` is create-if-absent and never overwritten, even with `--force`** — silently replacing a user's own agent instructions would be unforgivable for a trust product. The `.claude/settings.json` hook is *merged* in (other hooks and settings preserved, never duplicated, and a file we can't parse is left untouched and reported). To keep the guardrails honest, the rules pack's per-check list is **generated from the live check registry**, so it can't drift from what `revivify check` actually enforces. Non-destructiveness isn't a nicety here; it's the same "don't give false reassurance / don't damage what you touch" ethos as #10 and #12, applied to the filesystem.

### 16. Milestone acceptance is a hand-held UAT, driven by Claude
CI proves the code *works*; it can't prove the milestone is *the right thing* for a non-developer. So each milestone closes with a **User Acceptance Test**: a prescriptive, Claude-driven session where Jake runs one scenario at a time — exact command, exact expected output, ✅/❌ — ending in a signed pass/fail sheet, and the release tag is the reward for all-green ([`docs/uat/`](uat/)). It's the human half of the gate, and it's built for the same person the product is (no code-reading). From M4 the acceptance walkthrough lives *in each build ticket* (Gate 1 of [`build-gate.md`](build-gate.md)); the milestone roll-up sequences them. This is #6's "verify as the end user" habit turned into a repeatable, signed ritual — and it eats our own dog food: a quality-gate product that refuses to call a milestone done until a human has verified it.

### 17. Adopt Lucid's SDLC deltas — self-contained tickets, a build gate, and a wayfinder planning phase
Revivify's workflow (#13) was adapted *into* the sister project [Lucid](https://github.com/jakewisnieski/Lucid), which then grew useful additions worth bringing back. Three land here. **(a) Self-contained build tickets:** each milestone module is stamped from one template carrying not just Outcome + Acceptance criteria but **Grounded-in** (traceability to the FRs/decisions that justify it), an **Intent seed** (a pre-written `--intent` for `/no-mistakes` so a review can tell a deliberate choice from a mistake), the **Acceptance walkthrough** (its own UAT), and a **Close-out** runbook — so everything needed to build, prove, and merge a module lives in the ticket. **(b) A build gate** ([`build-gate.md`](build-gate.md)): "done" is a fixed two-gate close-out — Gate 0 automated (`/no-mistakes`), Gate 1 human (the walkthrough) — and Claude never self-merges. **(c) A wayfinder planning phase:** before code, product decisions are charted as tickets — a living `wayfinder:map` epic plus `Decide:`/`Research:`/`Prototype:` tickets grounded in primary sources, each resolved decision gisted into the map and, for load-bearing calls, into *this* log. That last point closes a real gap: the M3 decisions above (#14–#16) were made in conversation and are only now recorded — the wayfinder phase makes capturing them the default, not an afterthought. Adopted from **M4 onward**; M0–M3 stand as they shipped.

### 18. "Your call" clears the 10/10 bar as "resolved-and-visible" (M4)
Decision #9 set the ship-ready bar at a perfect 10/10 but explicitly deferred *how* a **"your call"** judgment item (e.g. brand-driven contrast) clears it. M4 settles it: your-call items get their **own resolution track**, separate from the objective pass/fail denominator in `score.ts`. The bar clears only when every objective check passes **and** every your-call item is resolved — **fixed, or knowingly accepted with a recorded reason.** An accepted item is **never scored as "passing"** (that would fake a green — the honesty line we hold in #10/#12) and **never silently dropped** like not-applicable (#9's promise is that *nothing broken is left behind*, so it must stay visible); instead it surfaces with its reason, e.g. `10/10 — 6 of 6 checks passing · 1 your-call accepted: "…"`. WCAG conformance is deliberately binary with no "accepted-risk" tier, but it *builds author judgment into* criteria like 1.4.3 Contrast (large-text / incidental / logotype exceptions), so keeping the judgment **visible and reasoned** is the honest mapping, not an escape hatch. The recorded reason comes from intent capture (#19). (Grilled in #25; drives build module M4.2.)

### 19. Intent is captured as a plain-language file + a structured accept-map (M4)
FR-6 wants page **intent captured before checking** so deliberate choices aren't flagged. The Stop hook and the own-the-fix loop run `check` **non-interactively**, so intent can't be an interactive prompt — it must be **persisted and read every run** (the AXI posture of #4). And intent serves two jobs with opposite needs: a **human narrative** and a **machine-checkable acceptance ledger.** So we split by fitness for purpose: narrative page intent lives in a plain-language **`.revivify/intent.md`** (scaffolded by `init` as a prompt-sheet, read by `check`, **optional** — no intent still runs, with a nudge; create-if-absent per #15), while your-call **acceptances** live as a structured **`accept:` map** (`rule-id: reason`) in `.revivify.yaml` that the bar (#18) consults. Plain language where a human writes; machine-readable where the bar reads. (Grilled in #26; drives M4.1 and completes #18's mechanism.)

### 20. The own-the-fix loop is emergent — Revivify never edits code (M4)
FR-9's loop is **propose → approve → apply → re-check.** The architecture is deliberately **CLI + hook, not MCP** (#4), and "the agent applies the fix" means the coding agent (Claude Code) **edits the files** — Revivify does not mutate the project. The pieces already exist: `check` emits structured findings with fixes and triage, and the Stop hook already **re-checks on "done"** (#14). So M4 adds **no new engine**; it makes the output *drive* the loop — the agent report groups findings into **actions** by triage (**well-fix-it** = an ordered fix plan the agent applies; **your-call** = a human-decision queue, never auto-applied; **just-so-you-know** = informational), and the human report **frames the approval** (the safe batch approved in one step, since "we'll fix it" is defined-safe; your-call decided individually — principle #4). A `revivify fix` view is optional. This is #6's own-the-fix loop and #4's "CLI + hook" made concrete. (Grilled in #27; drives M4.3.)

### 21. The M4 educational layer is per-finding cite + teach + verify (M4)
Principle #1 (**authority from standards**) and #4 (**teach as it goes**) require every finding to explain the *why* and cite the source. Today a finding carries a cited `standard` and a `detail` sentence; the **rich standalone report is explicitly fast-follow** (§8.2). So M4 keeps the educational layer as **per-finding metadata** and completes the trust loop *rule → why → source → verify*: keep the cited `standard`, **sharpen `detail` to teach user impact** (outcomes — "screen-reader users can't perceive this image" — not spec), and add a **`learnMore` URL** per rule to the exact published standard (WCAG Understanding / web.dev CWV / Search Essentials — **net-new**, since the catalog names standards but carries no links). URLs are asserted **from the rule registry** so guidance can't drift from the checks (the #15 pattern). (Grilled in #28; drives M4.4.)
