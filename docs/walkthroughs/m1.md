# M1 — the real engines (axe-core + Lighthouse)

**What this milestone proves:** Revivify's verdicts now come from **real, industry-standard engines**, not hand-rolled HTML checks. `revivify check` runs **Lighthouse** (which uses **axe-core** internally for accessibility), reports the actual category scores, and checks **13 cited rules** across accessibility, performance, SEO, and best practices.

**What changed from M0:** M0 read the HTML text for six basic things. M1 loads the page in a real headless browser, measures it, and judges it the way the professionals' tools do — so it can catch performance (load speed, layout stability), colour contrast, HTTPS, console errors, and image optimisation, which you can't see from the HTML alone.

**What's deliberately still out (honest limits):**
- **Non-text/UI contrast (WCAG 1.4.11)** and **broken outbound links** don't have a trustworthy automatic check yet — deferred to a later milestone rather than faked (see [decision log #10](../decision-log.md)).
- **HTTPS** shows as *not applicable* for a local page — a browser treats localhost as secure, so it can't be verified until the site is deployed ([decision log #12](../decision-log.md)). That's why a perfect local page reads "11 of 11," not 12.
- The viewport check reads as *not applicable* on a bare page with no `<head>` (a real Lighthouse quirk); on normal landing pages it works.

## Try it yourself (~2 minutes)

Setup once: `npm install`. The full audit launches Chrome, so each run takes **~30–45s** (you'll see a "Running the full audit…" note).

### 1. A perfect page → 10/10

```bash
npm run check -- ./examples/perfect
```

You should see a perfect score and the real Lighthouse category breakdown:

```
  Trust: 10/10 — 11 of 11 checks passing
  Lighthouse: Performance 100 · Accessibility 100 · Best Practices 100 · SEO 100
  …
  Ship-ready ✅  — a perfect 10/10. Nothing broken was left behind.
```

### 2. A raw page → real failures, with real scores

```bash
npm run check -- ./examples/starter-slop
```

```
  Trust: 6/10 — 6 of 10 checks passing
  Lighthouse: Performance 100 · Accessibility 56 · Best Practices 96 · SEO 73
  ✗ Images have alt text …
  ✗ Page declares its language …
  ✗ Page has a title …
  ✗ Page has a meta description …
```

### 3. The instant pre-check (no browser)

```bash
npm run check -- ./examples/starter-slop --fast
```

Runs only the six instant static checks — handy while iterating. It's a *narrower* look, so its score (2/10 here) isn't the real one; the full audit (6/10) is what certifies ship-ready.

### 4. The example gallery

Each page has a known full-audit result to compare against — see [`examples/README.md`](../../examples/README.md): `missing-alt` → 9/10, `no-meta-description` → 9/10, `accidental-noindex` → 9/10 (a "Your call").

## The 13 rules in M1

| Area | Rules | Engine / standard |
|---|---|---|
| ♿ Accessibility | images alt · text contrast · html lang · form labels | axe-core (in Lighthouse) — WCAG 2.2 |
| ⚡ Performance | LCP · CLS · image optimisation | Lighthouse — Core Web Vitals |
| 🔎 SEO | title · meta description · viewport · noindex | Lighthouse — Search Essentials |
| ✅ Best practices | HTTPS · console errors | Lighthouse — Best Practices |

The headline `X/10` and "N of M passing" are rule-based (10/10 = every applicable rule passes); the four category scores (0–100) are shown beside it as the "where it comes from" breakdown.
