# M4 — Check UX (see and drive the check→fix loop)

**What this milestone proves:** the cockpit isn't just a scoreboard anymore — it's where you *drive* the loop. You see the page's **intent** read back, findings sorted into **what I'll fix / what's your call / just so you know**, you **accept a judgment call right in the browser** and watch the trust dial clear the bar, and every finding links to the **published standard** so you can verify it yourself.

**Why it exists:** a non-dev shouldn't have to read CLI output — or hand-edit a YAML file — to run a quality gate. M4 brings the whole check→fix→accept loop into plain sight (PRD §10; decisions [#18–#23](../decision-log.md)).

**How it's built:** the engine (M4.1–M4.4) landed the logic in the check itself — intent capture, a your-call resolution track that never fakes a green, triage-as-actions, and `learnMore` links. M4.5 rendered it in the cockpit; M4.6 added the in-browser **Accept** control. This walkthrough is **cockpit-first** — you accept it by *looking*, not by running CLI scripts.

## Try it yourself (~3 minutes)

Jake runs Windows **PowerShell**, so this is PowerShell-first. From the repo root:

```powershell
# One-time in this window: allow npm/npx scripts to run.
Set-ExecutionPolicy -Scope Process -ExecutionPolicy Bypass

# Build a demo project: the "accidental noindex" page + a page intent + an empty accept map.
$DEMO = "$env:TEMP\revivify-m4-demo"
Remove-Item -Recurse -Force $DEMO -ErrorAction SilentlyContinue
New-Item -ItemType Directory -Force "$DEMO\.revivify" | Out-Null
Copy-Item .\examples\accidental-noindex\* $DEMO -Recurse
@"
# Page intent (for Revivify)
## What is this page for?
Waitlist signup for Bloom, a houseplant subscription.
## Deliberate choices a checker might flag
The hero uses our brand palette.
"@ | Set-Content "$DEMO\.revivify\intent.md" -Encoding utf8
"threshold: 10`nenforcement: warn`naccept: {}" | Set-Content "$DEMO\.revivify.yaml" -Encoding utf8

# Open the cockpit on the demo.
npx tsx src/cli.ts ui $DEMO
```

In the browser that opens:

1. Tick **fast pre-check** (instant, no Chrome) and click **Run check**.
2. You land on **9 / 10 · Not yet**, because one finding is a **your call** — a leftover `noindex` that hides the page from Google, which *can* be deliberate.
3. In the **Your call** card, click **Accept this — it's intentional**, type a reason (e.g. *"Staging site — hidden until launch."*), and **Confirm**.
4. Watch the dial **clear to 10 / 10 · Ship-ready**, while the item **stays visible** marked **accepted** with your reason. (Behind the scenes it recorded the reason in `.revivify.yaml` for you — no YAML editing.)
5. Open a **Learn more ↗** link on any finding — it goes to the exact WCAG / web.dev / Google standard.

Prefer the real thing? Untick **fast pre-check** to run the full Lighthouse audit (~30–45s) with the category gauges.

## What you should see — and why it's right

- **Your intent is read back.** The top card shows what you wrote in `.revivify/intent.md` (or nudges you to add it) — so deliberate choices aren't treated as mistakes ([#19](../decision-log.md)).
- **Findings are sorted into actions.** The violet **Plan** card frames the safe *"we'll fix it"* batch as **one approval**; *your call* items are yours to settle and are **never auto-applied** ([#20](../decision-log.md)).
- **A "your call" is resolved, not faked.** Accepting keeps the item **visible with its reason** and never renders it as a green ✓ — the honest mapping, not a false pass ([#18](../decision-log.md)). The dial only reaches 10/10 once every objective check passes *and* every your-call is resolved.
- **You can accept it right here.** No hand-editing YAML; a reason is **required** (accepting without one is refused, in the UI and on the server).
- **Every finding is verifiable.** Cited standard + a plain-language *user-impact* reason + a **Learn more** link to the source ([#21](../decision-log.md)).
- **Colour has meaning.** Green = pass/go only; violet = brand/action (the Plan badge, the Accept button, the dial) ([#11](../decision-log.md)).
